MedChem – Smart Pharmacy Inventory System

Module 2: User Authentication

Project Documentation

1. Module Introduction

The User Authentication module is a core component of the MedChem Smart Pharmacy Inventory System. Its main purpose is to secure the application and ensure that only authorized users can access the pharmacy data. In a real pharmacy environment, protecting medical and sales records is very important, so a reliable login system is required before anyone can use the software.

This module provides a simple and effective login mechanism using Firebase Authentication. It requires the user to enter a valid email address and a password to gain access. Once the user's credentials are verified by the Firebase server, they are allowed to enter the main application areas such as the dashboard, inventory management, and sales records.

To improve the overall user experience, this module also handles session management. This means that if a user refreshes the page or closes the browser temporarily, they do not have to log in again. The system remembers their active session in the background. It also secures the application by automatically redirecting any unauthenticated visitors back to the login page.


2. Module Objectives

The primary objective of this module is to restrict unauthorized access to the MedChem system. We want to ensure that only registered staff members can view or modify the pharmacy inventory and sales data. This is achieved by creating a secure login interface where users must prove their identity before entering the software.

Another important objective is to provide clear feedback to the user during the authentication process. If a user enters an incorrect password or an invalid email format, the system should display a helpful error message. We also use a loading spinner to show the user that the login request is being processed in the background.

Finally, the module aims to keep the application flow smooth and user-friendly. We achieve this by managing the user's login session automatically. If the user successfully logs in, they are redirected to the dashboard. If they try to access a protected page without logging in, they are immediately redirected back to the login page.


3. Module Scope

The scope of this module is strictly limited to user login, logout, and basic session management using Firebase Authentication. It is designed to handle the core security requirement of verifying a user's email address and password before granting access to the system. The module covers the creation of the login interface and the background logic needed to communicate with Firebase.

This module also includes the implementation of protected routes across the application. It ensures that the dashboard, inventory, sales, expiry, and notification pages are completely hidden from unauthenticated users. The scope includes automatic redirection logic so that users are always taken to the correct page based on their login status.

Features such as user registration, password recovery, email verification, and role-based access are outside the scope of this module. The system assumes that user accounts are pre-created manually in the Firebase console by the administrator. The focus is entirely on providing a simple, working authentication flow for existing users.


4. Technologies Used

React.js
React is a popular JavaScript library used to build the user interface of our application. We used it because it allows us to create reusable components, like the login form and the navigation bar. By using React's functional components and hooks, we keep our code organized and easy to maintain.

Bootstrap 5
Bootstrap 5 is a CSS framework that helps us style our web pages quickly. We used it to design a responsive login page and a clean dashboard grid without writing too much custom CSS. Its built-in classes made it easy to add professional-looking buttons, input fields, and spacing.

Firebase Authentication
Firebase Authentication is a backend service provided by Google that handles user login securely. We used its email and password authentication feature to verify our users. It saves us from having to write our own complex security code and manages user sessions automatically.

React Router DOM
React Router DOM is a library that allows us to navigate between different pages in our React application. We used it to set up our application's page addresses, like "/login" and "/dashboard". It also helps us protect specific pages by redirecting users who are not logged in.

React Toastify
React Toastify is a small library used to show popup notification messages to the user. We integrated it to provide instant feedback, such as showing a green success message when a user logs out. It is also used to display red error messages if a user types the wrong password during login.


5. Folder Structure

To keep the code clean and easy to manage, we organized the User Authentication module into a specific folder structure inside the src/modules/auth/ directory. This modular approach helps us separate the visual design from the background logic.

src/modules/auth/ ├── components/ │ └── LoginForm.jsx ├── constants/ │ └── authConstants.js ├── context/ │ └── AuthProvider.jsx ├── hooks/ │ └── useAuth.js ├── pages/ │ └── Login.jsx ├── routes/ │ └── ProtectedRoute.jsx ├── services/ │ └── authService.js └── auth.css

components/: This folder contains smaller, reusable UI pieces. For example, LoginForm.jsx handles the actual email and password input fields and validation logic.

constants/: This folder holds standard values used across the module. authConstants.js stores things like error messages and route paths, so we don't have to type them manually every time.

context/: This folder contains AuthProvider.jsx, which connects to Firebase and tracks whether the user is logged in or not. It shares this session information with the rest of the application.

hooks/: This folder contains useAuth.js, a custom React hook that allows any part of our application to easily check if the user is currently authenticated.

pages/: This folder contains the main screen layouts. Login.jsx is the main page component that wraps our login form and displays the application logo and background.

routes/: This folder contains ProtectedRoute.jsx, a special component that acts like a security guard. It checks if a user is logged in before allowing them to view a specific page.

services/: This folder holds the code that talks directly to the database. authService.js contains the actual functions that send the login and logout requests to Firebase.


6. Module Architecture

The architecture of the User Authentication module follows a simple, one-way flow of information. It starts when a user opens the web application and is presented with the login page. The user interacts with the Login Form by entering their registered email address and password into the provided text fields.

Once the user clicks the login button, the form sends these details to the Firebase Authentication service. Firebase acts as the secure backend that checks if the provided email and password match a valid account in the system database. If the details are incorrect, the process stops, and an error message is returned to the user.

If the details are correct, Firebase successfully authenticates the user and creates an active session. The Session Management system within our application then detects this successful login. It saves the user's active status so they do not have to log in again if they refresh the page.

After the session is confirmed, the user is passed through the Protected Routes system. This acts as a final security checkpoint, confirming that the current visitor has permission to view the private pages. Once cleared by the protected route, the user is successfully redirected to the Dashboard.


7. Functional Requirements

Login
The system must allow existing users to log in by providing a valid email address and password. It must validate that the email is in the correct format and that the password is at least 6 characters long before sending the request. While the system is verifying the details, it must show a loading spinner and disable the login button to prevent multiple clicks.

Logout
The system must provide a clear logout button on the main dashboard for users to safely end their session. When the logout button is clicked, the system must immediately invalidate the user's active session using Firebase. After successfully logging out, the system must display a success message and redirect the user back to the login page.

Session Management
The system must be able to remember if a user is already logged in, even if they refresh the web browser or open a new tab. If a logged-in user tries to visit the login page again by typing the URL, the system must automatically redirect them to the dashboard instead. This saves time and provides a smooth user experience.

Protected Routes
The system must secure all internal pages, including the dashboard, inventory, sales, expiry, and notification pages. If an unauthenticated person tries to type the web address for any of these private pages directly into their browser, the system must detect that they are not logged in. It must then block their access and redirect them immediately to the login page.


8. Firebase Authentication Flow

The authentication flow begins when the user submits their email and password through the login form. The React application captures these details and uses a specific Firebase function called signInWithEmailAndPassword. This function sends the data securely over the internet to the Firebase servers for verification against the project's user database.

If Firebase successfully verifies the credentials, it returns a special user object back to the React application. Our application uses a Firebase listener called onAuthStateChanged, which continuously runs in the background. As soon as the user logs in, this listener detects the change, updates the application's global state to show that the user is authenticated, and removes the loading screen.

When the user is done using the application and clicks the logout button, the flow reverses. The React application calls another Firebase function named signOut. This tells the Firebase server to terminate the active session. The onAuthStateChanged listener immediately detects that the user object is now empty, updating the application state to unauthenticated and returning the user to the login page.


9. Validation Rules

Email Validation
Before the system attempts to log the user in, it first checks if the email field is empty. If it is empty, it stops and prompts the user to enter their email. Next, it uses a regular expression to verify that the text entered actually looks like a real email address (for example, containing an "@" symbol and a domain like ".com"). If the format is invalid, a red warning text is shown under the input box.

Password Validation
Similar to the email field, the system first checks if the password field is left completely blank. If the user has typed a password, the system then checks its length. Based on Firebase requirements, the system enforces a rule that the password must be at least 6 characters long. If it is shorter, an error message is displayed, and the form will not submit.

Error Handling
If both inputs pass the local checks but the login still fails (for example, if the user typed the wrong password), Firebase returns an error code. Our application catches this specific error code and translates it into a simple, human-readable message like "Incorrect password. Please try again." This message is then shown to the user as a popup notification at the top of the screen.


10. User Interface

Login Page
The login page features a clean, professional design built with Bootstrap 5. It is centered on the screen over a dark blue gradient background. A white card holds the MedChem logo, system title, and input fields for the email and password. A small eye icon is provided next to the password field, allowing the user to show or hide their typed password. When the user clicks the "Sign In" button, a loading spinner replaces the icon to show that the system is processing the request.

Dashboard after Login
After a successful login, the user is greeted by the main dashboard. This page uses a light gray background with a dark top navigation bar. In the center, a welcome message is displayed. Below the welcome message, there is a clean grid of four cards representing the main modules: Inventory, Sales, Expiry, and Notifications. Each card features a descriptive icon and a button, making it very easy for the user to navigate the software.

Logout Button
The logout functionality is integrated directly into the dashboard's top navigation bar for easy access. It is designed as a red button positioned in the top-right corner, ensuring it is always visible but out of the way of the main content. When the user clicks this button, the system logs them out immediately, displays a green success popup message, and returns them to the secure login page.

13. Advantages

One of the main advantages of this authentication module is its high level of security. By using Firebase Authentication instead of building a custom login system from scratch, the application relies on Google's secure servers to handle sensitive user passwords. This reduces the risk of data leaks and ensures that the pharmacy's information remains protected at all times.

Another major advantage is the smooth user experience provided by React. Because the login page performs validation instantly on the client side, users are immediately informed if they type an invalid email or a short password. They do not have to wait for the page to reload to see their mistakes, which saves time and frustration.

Finally, the implementation of protected routes adds a strong layer of privacy to the system. It guarantees that no one can bypass the login screen by simply guessing the web address of the dashboard or inventory pages. The automatic redirection ensures that the application always behaves predictably, keeping unauthorized visitors out while allowing logged-in staff to work without interruption.


14. Limitations

Currently, the biggest limitation of this module is that it does not have a user registration page. New staff members cannot create their own accounts through the application interface. Instead, the system administrator has to manually add their email and password directly into the Firebase console, which can be time-consuming if the pharmacy hires new staff frequently.

Another limitation is the lack of a password recovery system. If a user forgets their password, they cannot reset it on their own using a "Forgot Password" link. They must contact the system administrator to manually send a password reset email through the Firebase backend, which causes delays in their daily work.

Finally, the system currently treats all logged-in users equally. There is no role-based access control implemented yet. This means that any staff member who logs in has full access to every module, including sensitive areas like sales and inventory editing. Ideally, a regular cashier should not have the same administrative permissions as the pharmacy manager.


15. Future Improvements

To make the application more user-friendly, a "Forgot Password" feature should be added in the future. This would allow users to securely reset their own passwords by receiving an automated email link from Firebase. It would reduce the dependency on the system administrator for basic account recovery tasks.

Another planned improvement is the implementation of Role-Based Access Control. By assigning specific roles like "Admin" or "Cashier" to different accounts, the system could restrict certain actions. For example, only an Admin would be able to delete stock from the inventory, while a Cashier could only view the stock and record sales.

Finally, we could improve the login experience by adding alternative authentication methods, such as Google Login. This would allow staff members to sign in quickly using their existing Google accounts with just one click, removing the need for them to remember a separate password for the pharmacy system.


16. Conclusion

In conclusion, the User Authentication module successfully fulfills its primary goal of securing the MedChem Smart Pharmacy Inventory System. By integrating Firebase Authentication, we built a reliable and functional login system without overcomplicating the underlying code. The implementation effectively ensures that only registered pharmacy staff can view and manage sensitive inventory and sales records.

The use of React and Bootstrap allowed us to create a clean, responsive, and user-friendly interface. Features like instant form validation, clear error messages, and loading indicators provide a professional experience for the end user. Furthermore, the session management and protected route logic ensure that the application behaves correctly, keeping unauthorized users out of the private dashboard sections.

Overall, this module lays a strong and secure foundation for the rest of the software project. While there is room for future improvements like role-based access and password recovery, the current implementation meets all the basic security requirements for a final year university project and is ready to support the daily operations of a small pharmacy.
